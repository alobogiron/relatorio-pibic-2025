Tendo como principal objetivo, comunicar ao sistema a entrada ou saída de qualquer indivíduo em uma instalação que possua um dispositivo BIoT. Comumente, os sistemas de software IoT baseiam-se em séries temporais

A abordagem proposta anteriormente, para monitorar a quantidade de indivíduos em umas instalação.

\begin{algorithm}[t]
\caption{Pattern Extraction and Matching from Time-Series Tuples}
\label{alg:pattern_match}
\begin{algorithmic}[1]

\Function{BINARIZE}{$X,\;\theta$}
  \State \Comment{$X$ is an array of tuples, e.g., $X[i] = (x_1,x_2)$}
  \State $B \gets [\ ]$
  \For{$i \gets 1$ \textbf{to} $|X|$}
    \State $(u,v) \gets X[i]$
    \State $b_1 \gets \begin{cases} 1, & u \ge \theta \\ 0, & u < \theta \end{cases}$
    \State $b_2 \gets \begin{cases} 1, & v \ge \theta \\ 0, & v < \theta \end{cases}$
    \State \Call{APPEND}{$B,\;(b_1,b_2)$} \Comment{Binary tuple}
  \EndFor
  \State \Return $B$
\EndFunction

\vspace{0.4em}

\Function{DEDUP\_CONSECUTIVE}{$B$}
  \State \Comment{Removes repeated values only when they are consecutive}
  \If{$|B| = 0$}
    \State \Return $[\ ]$
  \EndIf
  \State $P \gets [B[1]]$ \Comment{Output pattern}
  \For{$i \gets 2$ \textbf{to} $|B|$}
    \If{$B[i] \ne B[i-1]$}
      \State \Call{APPEND}{$P,\;B[i]$} \Comment{Keep only changes}
    \EndIf
  \EndFor
  \State \Return $P$
\EndFunction

\vspace{0.4em}

\Function{MATCH\_PATTERN}{$P,\;E,\;S$}
  \State \Comment{$P$ is the deduplicated pattern; $E$ and $S$ are predefined patterns}
  \If{\Call{EQUAL}{$P,\;E$}}
    \State \Return \textbf{ENTERED} \Comment{Matches entry pattern}
  \ElsIf{\Call{EQUAL}{$P,\;S$}}
    \State \Return \textbf{LEFT} \Comment{Matches exit pattern}
  \Else
    \State \Return \textbf{NO\_MATCH}
  \EndIf
\EndFunction

\end{algorithmic}
\end{algorithm}

\subsection{Análise dos Dados Coletados}
\label{subsec:data_analysis}

Os trabalhos desenvolvidos partem do contexto já existente do projeto, tendo em vista o nível de maturidade do sistema, bem como todas as decisões arquiteturais anteriores. Sendo assim, esta seção compreende as melhorias implementadas acerca do projeto já proposto, o qual é apresentado na Figura 1.

\newpage

\begin{figure}[ht]
    \center
    \includegraphics[scale=0.3]{images/arq_safe.png}
    \caption{Esquemático simplificado do projeto SAFE-UFRJ.}
    \label{fig:arq_safe}
\end{figure}

O sistema existente, é constituído por um dispositivo IoT que realiza as medições acerca do ambiente e publica as informações via \textit{broker} através do protocolo MQTT(\textit{Message Queuing Telemetry Transport}), a ferramenta utilizada para a comunicação é o RabbitMQ. As especificações do dispositivo, consistem em um ESP32 com FreeRTOS para o escalonamento das tarefas, que faz uso dos sensores CCS811(dióxido de carbono e compostos voláteis), DHT11(temperatura e umidade) e dois HC-SR04(controle do fluxo de pessoas), para realização das medidas.

Ademais, o \textit{backend} consome estes tópicos publicados pelo dispositivo, transforma as informações em dados mais tangíveis ao usuário e os armazena no banco de dados relacional(PostgreSQL), bem como disponibiliza rotas de uma API(\textit{Application Programming Interface}) para o funcionamento da interface de gerenciamento de usuários e instalações do projeto SAFE-UFRJ(\textit{ManagerSAFE}). Foram utilizados respectivamente Typescript e Express para a construção do \textit{backend}, bem como o \textit{dashboard} foi desenvolvido em Javascript.

Assim, um dos primeiros trabalhos desenvolvidos como bolsista consistiu-se na evolução do código do dispositivo de acordo com o diagrama presente no Apêndice A, para garantir melhor sincronicidade entre as tarefas, tendo em vista que para sistemas operacionais em tempo real, como o FreeRTOS, este é um fator crucial para o funcionamento. 

Também, foram implementados classificadores de erro, para identificar medidas que foram realizadas com insucesso ou que aconteceram durante o mau funcionamento de algum componente do dispositivo, comprometendo a medição. Para isto, foram construídos tipos de erro, como o apresentado na Figura ~\ref{fig:ex_erro}, permitindo a distinção entre medições corretas e insucessos.

\newpage

\begin{figure}[ht]
    \center
    \includegraphics[scale=0.4]{images/exemplo_erro.png}
    \caption{Exemplo de tipo de erro.}
    \label{fig:ex_erro}
\end{figure}

O tipo de erro é classificado em 64 \textit{bits} e cada casa em sua representação binária equivale a um tipo de erro único. Assim, caso o dispositivo tenha insucesso apenas na coleta de temperatura, o identificador que deve ser inserido na variável de erro é 0000010. Esta checagem é realizada em todas as etapas da medição, bem como na obtenção do registro de horário da coleta, como presente no diagrama do Apêndice A. 

Após isso, a ênfase se voltou para a visualização das informações coletadas. Como apresentado na Figura ~\ref{fig:arq_safe} o \textit{dashboard} coleta informações diretamente do \textit{broker} para apresentá-las ao usuário. No entanto, a exibição das medidas de maneira bruta, assim como são coletadas pelos sensores, não apresenta tanto potencial de tomada de decisão ao usuário.

Desta forma, o próximo trabalho desenvolvido durante a vigência da bolsa PIBIC compreende a implementação de rotas para a API já existente, possibilitando o consumo das medições armazenadas no banco de dados de maneira assíncrona e apresentando informações de maior valor agregado ao usuário via \textit{dashboard}. Nesta etapa, foram desenvolvidas rotas tanto para o consumo das informações à respeito das instalações, bem como a apresentação das medidas relacionadas a cada dispositivo ou instalação, facilitando possíveis testes futuros.

A Figura ~\ref{fig:rotas_api}, apresenta um recorte da documentação da API contendo as rotas implementadas, sendo elas encarregadas pela listagem de instalações e a listagem de medidas.

Por fim, a última etapa desenvolvida durante este período, abrange a adequação do \textit{dashboard} do sistema para o consumo das rotas desenvolvidas e apresentação das medições. Para isto, foi utilizada a estratégia de \textit{pooling} para o consumo da API, a qual consiste em realizar consultas periódicas para alimentação dos componentes presentes no \textit{dashboard}, logo as informações presentes na interface são atualizadas a cada 15 segundos. 

Desta forma, cada instalação é apresentada como um retângulo, contendo a identificação, localização, medidas de biossegurança e o registro de hora da última medida, assim como apresentado na Figura ~\ref{fig:dashboard_1}.

As informações apresentadas em cada retângulo, correspondem as medidas coletadas pelo dispositivo presente naquela instalação. Como considera-se que há apenas um dispositivo por sala (instalação), não se faz necessário a adoção de técnicas para unir informações de dois dispositivos. 

Ademais, o usuário pode obter mais informações acerca da instalação desejada apenas selecionando o respectivo retângulo. Para isto, foi implementada a tela de Detalhes da Instalação apresentada na Figura ~\ref{fig:dashboard_2}, a qual apresenta os relatórios de cada propriedade da medição.

Os gráficos presentes nesta tela, apresentam as medições realizadas, bem como o registro de horário de cada medição, permitindo ao usuário observar variações e padrões nos indicadores de cada instalação. As barras verticais presentes nos gráficos, correspondem aos erros de medição descritos anteriormente, caso o dispositivo detecte algum problema durante a realização da medida, o mesmo realiza a publicação do tópico, mas adiciona um tipo de erro ao pacote, bem como altera o valor da medida que foi realizada com insucesso, para um valor fora do escopo de leitura dos sensores e consequentemente invalidando a medida.
